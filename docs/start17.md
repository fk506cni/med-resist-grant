## 言語・可読性ルール（レポート執筆時に必ず守ること）

報告書はエンジニア以外も読みます。以下のルールを守って書いてください。

- **対象読者はエンジニアに限らない。** 専門知識のない関係者でも読み解ける表現を選ぶこと。
- 文体は **常体（だ・である）または敬体（です・ます）のいずれかに統一**。混在禁止。

### 英語のまま使ってよいもの

- ファイルパス、ディレクトリ名、コマンド名（`scripts/build.sh`、`docs/report17.md` 等）
- エラークラス名・関数名・変数名・テーブル名・カラム名（`_ensure_table_rows`、`SCHEMA_VERSION` 等）
- 業界で定着した略語: API, CLI, UI, JSON, YAML, SQL, DDL, MCP, PR, CI, OS, URL, ID, SHA, regex, lint, hook, commit, branch, merge, diff
- バージョン番号、識別子（`v4`、`R36`、`C17-01`、`TD-012` 等）

### 日本語に置き換えるもの

| 避ける英語 | 推奨される日本語 |
|---|---|
| carry-over | 継続審議 / 持ち越し |
| rolling review | 継続レビュー |
| fabrication | 記述の捏造 / 実体のない記述 |
| sync miss | 同期漏れ |
| dogfood readiness | 自社利用準備 / 試運転準備 |
| blocker | 阻害要因 |
| consistent / inconsistent | 整合している / 整合していない |
| specific patterns | 特定のパターン |
| go / no-go | 実施可 / 実施不可 |
| escape hatch | 回避策 |
| cross-cutting | 横断的な |
| typo | 誤記 |

このリストに無くても、**英単語を組み合わせた造語的フレーズは原則禁止**。日本語化するか、定義を併記して使う。

### 略語の扱い

- 上記「定着した略語」以外を使う場合は、**初出時に日本語補足を併記**（例: `PCRE（Perl 互換正規表現）`、`ERE（拡張正規表現）`）。
- 同レポート内で 2 回目以降は省略可。

## 重大度の基準（提出前最終チェック向け）

- **Critical**: 成果物として致命的（誤りを含む / 公開できない品質 / 採点者が即座に減点する事項）

- **Minor**: 細部の修正だが提出までに直したい
- **Info**: 次回以降の改善事項（提出後の中期課題、本提出には影響しない）
